📚 Labs 1–5

Laboratory Work

Five progressive labs building from static layouts through to persistent, database-backed apps.

1

Basic Layout (UI)

Row · Column · Box · Image · Text

Building a static screen using Row, Column, Box, Image, and Text composables in Jetpack Compose to establish a structured UI foundation aligned with SDG 4.

Lab 1 taught me how Compose uses Row and Column to arrange UI. I reverse-engineered a reference app which is Jungle for SDG 4 by recreating its layout with my own color scheme. At first the nesting of composables was confusing, but once I understood the tree structure it clicked.
2

Interaction

TextField · Button · mutableStateOf · remember

Adding TextField, Button, and state management to make the UI respond to user input and persist values across recompositions.

Lab 2 introduced me to state management in Compose and I learned that remember holds state during recomposition and mutableStateOf triggers a redraw when the value changes. I added interactive name entry and a button to my SDG app. The hardest part was understanding why state needs to be in the composable, not outside it.
3

Material Design

Theme · ElevatedCard · animateContentSize()

Applying a custom Material 3 theme generated from Material Theme Builder, using Card composables and content-size animations for a polished look.

Lab 3 showed me how professional apps get their consistent look with a central color scheme defined in Color.kt that flows through every composable automatically. I used Material Theme Builder to generate a blue palette matching SDG 4 and wrapped my content in ElevatedCards. Adding expand/collapse animation with animateContentSize() made the UI feel noticeably more polished.
4

Navigation & ViewModel

NavController · Routes · Shared State

Building a multi-screen app with NavController, enum-based routes, and a shared ViewModel so state persists across all screens.

Lab 4 transformed my single-screen app into a real multi-page experience. I defined routes in an enum and used NavController to navigate between them. The ViewModel was the key insight as it lives outside any one screen, so data like the user's name doesn't reset when you navigate away. This is the foundation that Focus Buddy is built on.
5

Room Database

Entity · DAO · StateFlow · MVVM

Adding persistent local storage using Room — Entity, DAO, Database, and StateFlow — to complete the full MVVM architecture with a live-updating UI.

Lab 5 completed the MVVM architecture by adding a real database layer. I learned that Room sits between the ViewModel and the actual SQLite file — the DAO defines what operations are possible, and Room generates the SQL automatically. The biggest concept shift was understanding Flow: instead of reading data once, the DAO returns a live stream that updates the UI automatically whenever the database changes.
🚀 Projects

Projects

Two Android apps applying everything learned across Labs 1–5, built around SDG 4: Quality Education.

Project 1
Focus Buddy — Core
Lab 4 + Working Features
Problem Statement Many students struggle to build consistent study habits due to a lack of motivation and structure. Focus Buddy gamifies the study experience through a virtual pet system and XP rewards to keep learners engaged.

Features: Home & Dashboard screens, study scheduling, subject quizzes, flashcards, Pomodoro timer, and a pet growth system across 7 screens powered by NavController and a shared ViewModel.

Project 1 was my first time building a complete multi-screen Android app from scratch. Wiring the ViewModel to every screen taught me how a single source of truth prevents data inconsistency. The gamification mechanic (XP → pet level) gave the app a clear reason to exist, which made design decisions much easier.

Project 2
Focus Buddy — Full Stack
Lab 5 + API + Sensors + Cloud
Problem Statement Students need not just structure, but also environmental feedback, live motivation, and community accountability to sustain studying. Focus Buddy v2 adds real-time cloud sync, hardware sensors, and a live API to address these gaps.

New in v2: Room Database for offline persistence, Firebase Firestore for shared community goals, ZenQuotes API via Retrofit for daily motivation, accelerometer shake-to-pause, and light sensor study environment feedback.

Project 2 was where everything came together — Room kept data alive across sessions, Retrofit fetched live quotes, Firebase shared goals across devices, and the accelerometer responded to physical movement. Integrating hardware sensors was the most exciting part; it made the app feel physically aware rather than just a digital checklist.

💭 Reflection

Learning Journey

From static layouts in Lab 1 to a cloud-connected, sensor-aware app in Project 2.

Starting from Lab 1's static layouts, each lab progressively added a new architectural layer — interactivity, theming, navigation, and finally persistence — until by Project 2 I had built a fully connected, hardware-aware app. The biggest challenge was internalizing the separation of concerns: screens should only call ViewModel functions, never touch Room or Firebase directly. Project 2 was where everything came together — Room kept data alive, Retrofit fetched live quotes, Firebase shared goals across devices, and the accelerometer responded to physical movement. I now understand why professional apps are built in layers: each layer can be changed independently without breaking the rest.
